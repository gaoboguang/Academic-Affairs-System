# AGENTS.md

本仓库用于开发“本地教务工具”，目标是构建一个面向高中学校场景的本地单机系统。

## 你开始任何工作前必须先做的事

1. 阅读 `docs/local_edu_tool_dev_spec.md`。
2. 按文档中的里程碑顺序工作，不要一开始试图实现所有模块。
3. 若发现文档中有实现细节缺失，可在不偏离目标的前提下做合理补充，但不得改变核心定位：**本地、单用户、离线优先、中文后台系统**。

## 强制约束

1. 仅做本地运行版本，默认监听 `127.0.0.1`。
2. 不引入必须联网才能运行的第三方在线服务。
3. 数据库存储使用 SQLite。
4. 关键业务规则必须配置化，禁止硬编码在页面中：
   - 成绩阈值
   - 冲稳保区间
   - 艺体推荐规则
   - 工作量系数
   - 班主任量化规则
5. 附件文件不得直接存入数据库，只保存本地路径和元信息。
6. 所有页面默认中文。
7. 优先兼容 Windows 开发和运行。
8. 每个里程碑都必须保证：
   - 项目可运行
   - 核心流程可演示
   - 核心逻辑有测试
   - README 已更新

## 优先级规则

### P0 必须先完成
- 工程初始化
- 基础数据中心
- 学生管理
- 教师管理
- 考试与成绩导入
- 学生/班级/教师成绩分析
- 课表导入
- 课时统计
- 教师工作量计算
- 备份恢复
- Excel 报表导出

### P1 再做
- 成长档案
- 院校/专业/录取库
- 普通生推荐
- 艺体生推荐
- 评教统计
- 班主任量化

## 代码结构要求

1. 使用 monorepo 结构。
2. 将前后端分开放在 `apps/frontend` 与 `apps/backend`。
3. `data/` 目录用于本地数据库、上传文件、模板、备份、导出和日志。
4. 业务逻辑按模块划分，不要把所有逻辑写进单个文件。
5. 至少分层为：models、schemas、repositories、services、analytics/importers/exporters。

## 实现策略

1. 先建立基础骨架与迁移系统。
2. 再做主数据与导入导出。
3. 再做分析计算。
4. 再做推荐与量化。
5. 再做体验优化和打包。

## 导入导出要求

1. 为学生、教师、成绩、课表、录取数据提供模板下载。
2. 导入失败必须输出错误报告。
3. 导入结果必须记录批次。
4. 导出文件统一放到 `data/exports/`。

## 测试要求

至少对以下逻辑编写单元测试：
- 排名与百分位
- 总分计算
- 优秀率/及格率
- 冲稳保分类
- 艺体推荐筛选
- 工作量计算
- 导入校验器

## 禁止事项

1. 不要把所有业务逻辑塞进前端。
2. 不要在没有迁移脚本的情况下直接改数据库结构。
3. 不要为了“快”而省略导入校验。
4. 不要在没有测试的情况下重构核心计算逻辑。
5. 不要优先做华而不实的 UI 动效。

## 完成定义

每个阶段完成时，必须满足：
1. 本地可以一键启动。
2. 能用示例数据演示当前阶段功能。
3. 核心测试通过。
4. README 中补充了当前阶段的运行说明和已实现范围。
