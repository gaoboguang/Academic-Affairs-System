## 0. 给 Codex 的最高优先级说明

你正在接管一个“本地教务系统”项目。用户没有编程基础，因此你不能把技术判断、代码修改、依赖安装、启动命令、报错排查、测试验证等问题交给用户解决。

你的职责是：

1. **先读取真实仓库代码**，不要假设技术栈、目录结构、接口路径、数据库结构。
2. **以仓库真实代码为准**，如果本文档的任务描述与仓库现状不一致，必须优先服从仓库代码。
3. **直接修改代码**，不要只给方案。
4. **运行必要命令**，包括安装依赖、启动服务、执行测试、构建检查、类型检查、lint 等。
5. **遇到报错先自主排查并修复**，不要把报错原样抛给用户。
6. **所有功能开发完成后必须给出中文总结**，让非程序员也能看懂。
7. **不得要求用户做技术决策**。如果遇到技术选型问题，请根据当前仓库已有技术栈自主选择最小改动方案。
8. **不要破坏已有功能**。每次修改前先理解相关文件，每次修改后运行验证。
9. **不要提交密钥、真实隐私数据、数据库密码、个人信息样例**。
10. **不要使用 Windows 作为主环境**。后续开发说明、脚本、命令和文档全部以 Mac 为主。

## 2. Mac 主开发环境原则

从现在开始，项目主开发环境固定为 Mac。

Codex 在开发时需要遵守：

1. 优先使用 Mac / zsh / Homebrew 相关命令。
2. 文档中不要再把 Windows 作为主要开发环境。
3. 可以保留必要的跨平台说明，但主流程必须是 Mac。
4. 如果发现仓库中存在仅适用于 Windows 的路径、脚本或说明，需要改为 Mac 优先版本。
5. 所有启动步骤应最终沉淀到：
   - `docs/mac-dev-setup.md`
   - `README.md`
   - `AGENTS.md`
6. 如果项目需要环境变量，必须创建或补全：
   - `.env.example`
   - 不要提交真实 `.env`
7. 如果项目有数据库初始化、迁移、种子数据，应在 Mac 文档中写清楚完整步骤。

## 17. 通用开发规则

所有 Codex 窗口都必须遵守以下规则。

### 17.1 读取真实代码

每个窗口开始时必须先读取：

- `AGENTS.md`
- `docs/repo-audit.md`
- `docs/development-roadmap.md`
- 与当前任务相关的源码文件

禁止不看代码直接开发。

### 17.2 最小改动原则

优先使用仓库已有：

- 框架；
- 数据结构；
- 命名风格；
- 组件；
- API 设计；
- 数据库访问方式；
- 测试框架。

不要为一个小功能大规模重构。

### 17.3 后端校验优先

涉及数据写入的功能，后端必须校验：

- 必填字段；
- 唯一性；
- 权限；
- 数据引用是否存在；
- 冲突；
- 重复提交；
- 文件上传格式。

前端校验只能作为用户体验，不能代替后端校验。

### 17.4 非程序员用户友好

所有错误提示应尽量用人话，例如：

不推荐：

```text
Foreign key constraint failed.
```

推荐：

```text
第 12 行导入失败：该学生所属班级不存在，请先创建班级或检查班级名称是否填写正确。
```

### 17.5 数据安全

禁止：

1. 提交真实密码；
2. 提交真实密钥；
3. 提交真实学生隐私数据；
4. 删除重要数据没有确认；
5. 导入失败还写入一半数据；
6. 静默覆盖已有成绩；
7. 只在前端做权限控制。

### 17.6 测试要求

每个窗口完成后至少执行一类验证：

1. 自动化测试；
2. 构建检查；
3. 类型检查；
4. lint；
5. 数据库迁移检查；
6. 手动业务流程验证。

如果某类测试无法运行，必须说明原因，并尝试修复。

## 24. 不建议的操作

不建议用户这样做：

1. 一开始同时开 10 个窗口。
2. 每个窗口都让它自由发挥。
3. 不等窗口 0 审计完就开发业务功能。
4. 多个窗口同时改数据库模型。
5. 多个窗口同时改登录权限。
6. 多个窗口同时改同一份前端路由。
7. Codex 报错后让用户自己处理。
8. 没有测试就合并。
9. 没有文档就继续扩展功能。
10. Windows 和 Mac 两套流程混着写。
