## 5. 为什么现在仍然建议 Windows 保持 3 个线程

### 默认建议：就 3 个，不增加
原因：

1. 当前 Windows 侧真正高优先级只有 C 和少量语义裁决
2. `DB RC1` 已冻结，当前不是大范围扩表期
3. 线程过多容易在同一个目录里互相踩文件
4. 3 个线程的分工最清晰：
   - 1 个主库单写
   - 2 个并行人工复核

### 第 4 个线程什么时候才允许开
只有同时满足以下条件才开：

1. Windows 项目已经在 Git 里
2. worktree 已经可用
3. 前 3 个线程边界已经稳定
4. 第 4 个线程只做单一冲刺任务

例如：

- 只做 `RC2` 打包验证
- 或只做 `2026 招生动态源清单` 的独立整理

否则，不开。

## 7. 本轮硬规则

### 7.1 主库单写规则
任何时刻，只有 `WIN_DB_CORE` 可以写：

```text
E:\shandong_admissions_2020_2025\data\local_edu_tool\local_edu.sqlite3
```

### 7.2 A/B 只能交付三类标准结果
必须输出：

- `patch.csv`
- `audit.md`
- `unresolved.csv`

可选输出：

- `evidence/`
- `notes.md`

### 7.3 Mac 本轮以 `DB RC1` 为只读接入基线
Mac 不要直接依赖现场工作库的瞬时状态。  
所有只读接入、驾驶舱显示、联调验证，默认都以 `DB RC1` 或 `DB_CORE` 明确发出的 handoff 包为准。

### 7.4 分片必须是确定性的
统一分片规则：

1. 优先 `college_id`
2. 若 `college_id` 为整数：
   - A 处理奇数
   - B 处理偶数
3. 若没有整数 `college_id`：
   - 使用 `college_code` 最后一位奇偶
4. 若还没有：
   - 按学校名称排序后按奇偶行号分片

### 7.5 schema 变更必须握手
凡是涉及：

- 表结构变化
- 字段变化
- 状态码变化
- 规则码变化
- Mac 侧接口契约变化

都必须：

1. 由 `WIN_DB_CORE` 在同步板登记
2. 说明影响范围
3. 标注是否影响 Mac
4. 只有冻结交接后才给 Mac 正式接入

### 7.6 不做这三件事
本轮明确不做：

1. 不做大范围目录重构
2. 不做 2026 全量招生数据大铺开
3. 不做脱离当前数据库边界的大改推荐引擎底层

## 10. 协作方式

### 10.1 Mac 可以依赖什么
Mac 可以依赖：

- 当前仓库已有逻辑
- `DB RC1`
- `WIN_DB_CORE` 明确给出的 handoff / summary / export / schema contract

### 10.2 Mac 不要依赖什么
Mac 不要依赖：

- 正在被 A/B 复核中的临时字段
- Windows 现场工作库瞬时状态
- 未写进同步板的结构变化

### 10.3 Windows 不要做什么
Windows 不要顺手做：

- Mac 主应用页面重构
- 导出层大改
- 推荐工作台复杂交互改造

## 12. `RC2` 的冻结定义

进入 `RC2` 前，至少满足：

1. `WIN_DB_CORE` 停止本轮主库写入
2. 已完成最终审计快照

## 13. Mac 接收 `RC2` 后做什么

Mac 拿到 `RC2` 后按这个顺序做：

1. 只读接入，不立刻改 schema
2. 驾驶舱联调
3. 推荐工作台联调
4. 打印 / 导出 / 报表摘要联调
5. 记录兼容问题
6. 只有必要时再发起 DB contract request
