# 本地教务工具交付版后续开发计划

- 文档日期：2026-04-24
- 当前目标：把项目从“可开发、可演示”推进到“个人可稳定投入使用”
- 当前业务焦点：生源地山东考生，全国高校在山东招生数据
- 交付形态：本地单机运行、SQLite 主库、中文界面、可备份恢复、优先兼容 Windows

## 1. 交付目标

本阶段不再追求全校多用户平台，也不优先扩其他省份。最终交付目标是：

1. 使用者可以在本机稳定打开系统，完成数据查看、学生维护、成绩分析、高考志愿筛选、志愿草稿、打印和导出。
2. 高考志愿模块以山东考生为核心，能消费全国高校在山东的招生计划、录取结果、分数线、一分一段、选科要求、招生章程和特殊类型规则。
3. 推荐结果必须说明数据来源、参考年份、录取口径、风险边界和人工复核点，不能把缺口数据包装成确定结论。
4. 数据库、迁移、导入、物化、备份和验证命令形成稳定流程，后续补数据不再靠临时脚本和口头记录。

交付版的核心判断标准不是“页面多”，而是“数据可信、流程闭环、输出可复核、启动可重复”。

## 3. 距离可投入使用的关键缺口

### 3.1 数据完整性缺口

当前最大短板仍是数据库，不是页面数量。

1. `admission_record` 目前只有普通类，特殊类型没有完整录取结果；对艺体、春考、综评、单招不能给出等同普通类的录取把握。
2. `enrollment_plan` 主要集中在 2025 年，2024 和 2020 数量很少；后续需要补齐近年山东招生计划，尤其是 2026 官方计划发布后的导入和比对。
3. `score_rank_segment` 当前只有 2024、2025 山东一分一段，若要更可靠地做历史分数到位次映射，应补齐 2020-2023。
4. `gaokao_score_line` 当前只有 2024/2025 的少量分数线；艺体、春考等特殊类型需要更完整的年度、批次、类别控制线。
5. `gaokao_policy_reference` 只有 4 条，政策链不足以支撑交付版的填报边界说明。
6. `gaokao_college_chapter_rule` 虽有 2052 条，但待人工复核状态仍占较大比例，章程限制链还不够可信。
7. 职业方向和专业映射已从空表变成可用基线，但仍偏宽，需要清洗高频专业，避免职业匹配解释过泛。

### 3.2 准确性缺口

- 目前部分结果依赖 fallback。fallback 可以提高可用性，但不能替代官方录取结果。
- 对山东特殊类型，系统必须继续显示“资格参考”“计划清单初筛”“需人工复核”等标签。
- 推荐结果需要继续强化“数据来源、参考年份、样本数、最近年份、是否跨年、是否院校线回退、是否规则基线”的结构化说明。
- 需要建立数据质量看板，把缺表、缺年、缺类型、缺章程、重复冲突和待审数量直接暴露出来。

### 3.3 交付体验缺口

- 启动链路已改善，但 Windows 一键启动、打包安装、数据目录定位、异常提示仍需最终验证。
- 高考数据页还需要从“只读驾驶舱”继续推进到“补数据、审数据、处理冲突、触发物化”的工作台。
- 规则表还缺更完整的前端管理页；特殊类型规则、赋分规则、选科字典已先补 `/recommendations` 只读页签和基线装载入口，但编辑维护仍需继续完善。
- 备份恢复虽然已有基础能力，但交付前必须做真实数据备份、恢复、二次启动验证。
- 需要一份面向使用者的简明操作手册，而不是只给开发命令。

## 4. 后续开发路线

### 阶段 0：交付安全底座

建议时间：2026-04-24 至 2026-04-26

目标：先保证项目能稳定启动、迁移、备份、恢复，后续补数据不破坏主库。

交付物：

- 固化 `npm run dev`、`start-local-edu.command`、后端迁移、数据物化、特殊类型规则初始化的使用说明。
- 新增或完善数据备份前置检查：执行并库、物化、规则 bootstrap 前自动提示备份位置。
- 为 `data/app.db` 增加数据健康检查命令，输出核心表数量、年份覆盖、考生类型覆盖、缺口摘要。
- 明确当前开发库、备份库、handoff 源库的关系，避免再次出现“哪个库才是主库”的混乱。

验收标准：

- 新机器或干净环境能按 README 拉起服务。
- `npm run backend:migrate` 可重复执行到 head。
- 数据补齐脚本执行前后都有备份和摘要。
- 启动失败时能明确提示端口、依赖或数据库问题。

2026-04-24 状态：

- 阶段 0 安全底座已补齐到可验收状态：`npm run dev` / `start-local-edu.command` / `backend:migrate` / `backend:data-health` / `backend:p0-check` 均已有 README 或 runbook 入口。
- `backend:merge-handoff`、`backend:materialize-gaokao`、`backend:bootstrap-special-types` 已具备写库前备份和执行前后健康摘要。
- `npm run backend:p0-check -- --json` 已验证过备份包、恢复库、SQLite 完整性和恢复库接口可用；该项通过只代表安全底座完成，不代表阶段 1 数据缺口已经补齐。

### 阶段 1：山东高考数据底座补齐

建议时间：2026-04-27 至 2026-05-05

目标：优先把“全国高校在山东招生数据”补到可用于个人填报辅助的程度。

交付物：

- 补齐山东近年招生计划：至少覆盖 2024、2025；2026 计划发布后作为单独批次导入。
- 补齐山东近年录取结果：普通类继续校验 2020-2025，特殊类型先确认官方可得范围，不能取得的明确标记为缺失。
- 补齐山东一分一段：至少覆盖 2020-2025，支撑分数到位次映射和跨年参考。
- 补齐山东省控线/批次线：普通类、艺术类、体育类、春考等按年度和类别结构化。
- 补齐山东政策参考：志愿数量、批次规则、投档模式、选科口径、专业调剂、特殊类型报名/校测/资格限制。
- 提升招生章程限制链：把语言、单科、性别、身高、视力、色觉、体检、校区、学费等字段继续审阅到可用状态。

验收标准：

- 高考数据页能按年份、类别、批次展示覆盖率，而不是只展示总数。
- 对任意推荐结果，系统能说明使用了哪些年份和哪些来源。
- 特殊类型缺完整录取结果时，页面和导出必须明确显示“无专门录取结果，只做初筛”。
- 数据导入后可以一键生成审计摘要，列出新增、更新、重复、冲突、待人工复核。

2026-04-24 状态：

- 阶段 1 的展示与审计能力已先落地：`npm run backend:data-health -- --json` 和 `/api/gaokao/data-health` 现在返回 `expected_years`、`missing_years`、按年明细、考生类型分布、批次/口径分布和 `audit_summary`。
- `/gaokao-data` 的“山东覆盖”页签已能展开查看按年份 / 类别 / 批次的覆盖矩阵，并展示数据导入审计摘要（当前记录数、疑似重复、冲突、待人工复核和缺口说明）。
- 当前真实主库不会伪造缺失数据：招生计划仍缺 2021-2023，录取结果普通类覆盖 2020-2025，特殊类型专门录取结果缺失并标记“仅可做初筛”，一分一段和省控线仍缺 2020-2023，政策参考仅 4 条，章程限制链待复核 1748 条。

### 阶段 2：山东志愿推荐交付版

建议时间：2026-05-06 至 2026-05-12

目标：让山东普通类推荐先达到可用，特殊类型达到“安全初筛 + 清晰复核”的可用水平。

交付物：

- 普通类推荐：以位次为主，分数为辅，输出冲刺、稳妥、保底、样本不足、参考偏旧等解释。
- 特殊类型推荐：继续区分省控线资格参考、计划清单初筛、普通类回退参考，禁止混成普通类录取把握。
- 志愿草稿：支持按山东批次和志愿数量规则检查上限、重复、保底不足、选科不符、章程待核。
- 输出材料：推荐报告、志愿草稿、Excel 导出、打印页统一显示风险边界。
- 规则管理：补一个面向山东的规则核对入口，能查看和维护 `province_volunteer_rule`、`province_score_transform_rule`、`subject_requirement_dict`、`special_type_rule`。
  - 2026-04-24 已先落地核对版：`/recommendations` 现有“省份规则 / 特殊类型规则 / 赋分规则 / 选科字典”入口，其中后三类规则按页签懒加载，支持列表、筛选和 bootstrap；完整编辑维护仍留到后续。

验收标准：

- 用一个山东普通类学生，可以完成“录入分数/位次 -> 生成候选 -> 保存草稿 -> 打印/导出”的完整流程。
- 推荐结果不能出现无来源、无年份、无样本数、无风险解释的条目。
- 特殊类型结果必须在页面、打印、Excel 三端一致显示初筛性质。
- 规则缺失、章程待核、参考年份偏旧、院校线回退等风险必须可见。

## 6. 开发执行顺序

接下来最建议按下面顺序推进：

1. 新增数据健康检查命令，先把当前数据库缺口自动列出来。
2. 把健康检查结果接到高考数据页，做“山东数据覆盖率”看板。

## 7. 质量红线

以下规则必须保留到交付版：

1. 不造数据。没有官方录取结果就不能生成看似确定的录取把握。
2. 不静默 fallback。凡是普通类回退、省控线参考、计划清单初筛、院校线回退、跨年参考，都必须在页面和导出中可见。
3. 不把规则硬编码回前端。规则应留在数据库、后端服务或可维护配置里。
4. 不直接覆盖主库。任何并库、物化、批量导入前必须先备份。
5. 不让数据来源丢失。导入批次、来源标题、来源链接、更新时间、审阅状态要尽量保留。
6. 不把“能跑”当作“可交付”。交付前必须验证启动、推荐、导出、备份恢复和离线运行。

## 8. 交付验收清单

交付前至少完成以下验收：

- `npm run backend:migrate` 通过。
- `npm run backend:data-health` 可输出核心表数量、年份覆盖、考生类型覆盖和 P0 缺口摘要。
- `npm run backend:p0-check` 可一次性完成数据健康、完整性、备份、恢复和恢复后启动验收。
- `npm run backend:materialize-gaokao` 可重复执行并输出摘要。
- `npm run backend:bootstrap-special-types -- --year 2025 --year 2026` 幂等通过。
- `npm run check` 通过。
- `npm run check:e2e` 至少覆盖首页、学生、成绩、推荐、志愿草稿、报表导出。
- 本地 `http://127.0.0.1:5173` 可访问，`http://127.0.0.1:8000/api/system/health` 返回正常。
- 真实主库可备份、可恢复、恢复后可启动。
- 山东普通类学生完整推荐流程跑通。
- 山东特殊类型学生初筛流程跑通，并明确显示不能替代正式录取结论。
- 打印页和 Excel 导出中的风险提示与页面一致。
- 用户手册覆盖启动、导入、推荐、备份恢复、常见问题。
