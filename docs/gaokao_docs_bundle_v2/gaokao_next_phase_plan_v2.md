## 5. 为什么要把“就业方向”做成正式模块

“按目标就业方向推荐”不能只理解为多加一个筛选框。

它本质上是：

- 从“未来想做什么”倒推“适合报什么专业”
- 再从“适合报什么专业”筛到“哪些院校在当前成绩范围内可报”
- 最终形成“职业方向 → 专业 → 院校/专业组/专业 → 志愿表”的完整链路

因此它必须同时作用于：

1. 就业知识库
2. 专业画像
3. 学生画像
4. 推荐引擎
5. 推荐解释层
6. 导出报告层

### 5.1 产品原则

1. **录取可行性优先，职业意向增强，不反客为主**
   - 先保证“能报、可报、风险可解释”
   - 再按就业方向和职业偏好做二次排序

2. **不直接把“职业”硬映射成“唯一专业”**
   - 一个职业方向通常对应多个专业路径
   - 一个专业也可能通向多个职业方向

3. **必须给解释，不只给结果**
   - 为什么推荐这个专业
   - 这个专业通常对应哪些就业方向
   - 是否需要读研/考公/考证/长周期培养
   - 当前数据的粒度和可靠性如何

4. **必须保留人工修正空间**
   - 就业方向与专业映射不是静态真理
   - 支持学校自己调整映射与权重

## 6. 功能板块规划

## 6.1 模块 A：志愿数据中心

### A1. 院校库（沿用并增强）

- 办学性质

- 是否艺体相关
- 选科要求原文
- 体检限制提示
- 就业方向摘要
- 升学/考证提示
- 备注

### A3. 招生计划库（本轮重点）
按“年份 + 省份 + 批次 + 院校 + 专业/专业组”维护。

最少字段：
- 年份
- 省份
- 批次
- 科类/模式
- 院校
- 院校代码
- 专业组编号（可空）
- 专业
- 专业代码
- 计划人数
- 选科要求
- 学费
- 学制
- 培养地点
- 学生类别
- 数据来源
- 导入批次

### A4. 历年录取库（沿用并增强）
建议增强字段：
- 数据粒度（院校级 / 专业级 / 专业组级）
- 投档最低位次
- 最低分
- 平均分
- 最高分
- 计划人数（可空）
- 征集志愿标记
- 数据可靠性等级
- 数据来源说明

### A5. 招生章程索引
先做索引，不优先做全文抓取。

建议字段：
- 年份
- 院校
- 标题
- 官方链接
- 是否已核验
- 风险摘要
- 最近核验时间

### A6. 职业与就业方向知识库（本轮新增）
这是新增的核心数据底座。

建议包含：
- 就业方向名称（如职业方向/岗位方向/行业方向）
- 上级分类（行业 / 岗位族 / 公共服务 / 技术研发 / 运营管理等）
- 方向说明
- 常见工作职位类别
- 常见单位行业
- 是否偏向继续深造
- 是否常要求资格证/执照
- 是否常要求考公/考编/规培/司法考试等长路径
- 风险提示
- 数据来源

注意：
- 这里不要求一开始就做很细的全国职业标准库
- v1 只要先支持“可维护、可映射、可在推荐中使用”即可

## 6.2 模块 B：专业-就业画像与映射层（本轮新增重点）

### 目标
把“专业”和“未来就业方向”之间的关系做成可维护、可解释的数据层。

### B1. 专业就业画像
最少支持以下维度：
- 专业 / 专业类
- 常见就业行业
- 常见职位类别
- 薪酬指数（可选）
- 专业匹配度/能力匹配度/职业发展满意度等评价指标（可选）
- 就业率 / 考研方向（若有）
- 是否适合直接就业
- 是否常见读研后竞争力明显提升
- 数据年份
- 数据来源
- 数据粒度（专业 / 专业类 / 学科门类）

### B2. 就业方向与专业映射
建立多对多映射，不允许一对一硬绑死。

建议字段：
- 就业方向
- 专业
- 关联强度（核心相关 / 强相关 / 一般相关 / 可转化）
- 推荐说明
- 是否需要读研增强
- 是否需要执业资格/证书
- 是否适合艺体路线
- 是否启用
- 人工备注

### B3. 学校自定义修正规则
考虑到不同学校和老师对“专业 → 就业路径”的理解会不同，必须允许：
- 手工新增映射
- 手工禁用映射
- 调整关联强度
- 添加本地备注

## 6.4 模块 D：学生志愿工作台

### 目标
围绕单个学生形成完整志愿工作流。

### 页面结构建议
1. 基本信息
2. 高考成绩与位次
3. 选科组合
4. 偏好设置
5. 职业与就业方向意向
6. 候选池
7. 志愿表编辑区
8. 风险校验区
9. 导出与保存区

### 新增输入（就业方向相关）
- 目标就业方向（支持 1~3 个）
- 目标行业偏好
- 目标岗位类别偏好
- 城市/地域就业偏好
- 是否接受“先读研再就业”路径
- 是否接受“考公/考编/资格证”路径
- 对薪酬、稳定性、发展空间的偏好权重（可简化为高/中/低）

后续如果需要，再接兴趣测评或职业倾向测评结果。

## 6.5 模块 E：推荐引擎升级

### 目标
从“按位次生成结果”升级到“按可报性过滤 + 录取风险分层 + 就业方向适配排序”。

### E1. 推荐流程

第一步：先过滤
- 省份
- 年份
- 批次
- 学生类别
- 艺体方向
- 选科要求
- 黑名单 / 白名单
- 民办 / 中外合作 / 高收费 / 地域限制

第二步：再做录取基线判断
- 最近一年最低位次
- 近三年最低位次中位数
- 近三年加权位次
- 样本不足提示
- 计划波动提示

第三步：再做职业方向匹配
- 目标就业方向与专业映射匹配
- 目标行业与专业就业画像匹配
- 目标岗位类别与专业就业画像匹配
- 是否接受读研/考证/长路径

第四步：再分层
- 冲
- 稳
- 保
- 高风险不推荐

第五步：再排序
- 默认先按录取风险层级排序
- 同层级内按就业方向匹配度、专业偏好、院校偏好排序
- 支持人工拖拽改序

### E2. 新增的推荐解释能力
每条结果至少要能解释：
- 为什么入选
- 用了哪几年的录取数据
- 依据的是位次还是分数
- 对应哪些目标就业方向
- 这个方向和专业是“核心相关”还是“可转化相关”
- 是否需要读研/考证/长周期培养
- 是否存在样本不足、计划波动、章程待核验等风险

### E3. 新增输出视图
1. 录取优先视图：按冲稳保展示
2. 就业方向视图：按目标就业方向展示
3. 专业路径视图：
   - 首选专业路径
   - 相近替代专业路径
   - 曲线进入路径
4. 风险视图：
   - 数据不足
   - 计划波动
   - 读研依赖
   - 资格证依赖
   - 章程待核验

### E4. 推荐策略原则
- **录取可行性是硬门槛**，不满足选科/批次/规则的不能因为就业方向匹配就强推
- **就业方向用于二次筛选和排序增强**，不是替代冲稳保逻辑
- **优先做可解释规则，不优先做黑箱综合分**

## 7. “按目标就业方向推荐”的详细需求拆分

## 7.1 需求目标

输入学生希望未来从事的就业方向，系统能反向给出：

1. 更适合的专业方向
2. 与目标方向核心相关的专业
3. 可作为替代路径的专业
4. 在当前成绩与位次范围内可报考的院校/专业组合
5. 相关风险提示与培养路径说明

---

## 7.2 子需求拆分

### 子需求 1：就业方向标准化
先把“就业方向”做成标准对象，不直接用自由文本硬搜。

建议最小分类：
- 技术研发类
- 医药健康类
- 教育教学类
- 法政公共服务类
- 财经管理类
- 传媒设计类
- 制造工程类
- 农林生态类
- 艺术体育类
- 服务运营类
- 数字化与新职业类

支持：
- 分类
- 别名
- 说明
- 是否需要长培养周期
- 是否适合艺体路线

### 子需求 2：专业-就业方向映射
系统必须能回答：
- 哪些专业是这个就业方向的首选专业
- 哪些专业是相近专业
- 哪些专业是曲线进入路径

因此至少维护：
- 关联强度
- 推荐说明
- 读研依赖
- 考证依赖
- 适合学生类型

### 子需求 3：学生职业意向采集
在学生工作台新增：
- 首选就业方向
- 次选就业方向
- 可接受替代方向
- 是否优先稳定性 / 薪酬 / 兴趣 / 长远发展
- 是否接受读研 / 考公 / 考证
- 目标就业城市（可选）

### 子需求 4：推荐引擎排序增强
现有推荐结果生成后，再按以下维度重排：
- 就业方向匹配度
- 专业路径强度
- 城市就业偏好（可选）
- 升学/证书路径可接受度

### 子需求 5：结果解释增强
结果页增加解释块：
- 该专业为何匹配目标就业方向
- 常见就业行业
- 常见职位类别
- 是否更适合读研后发展
