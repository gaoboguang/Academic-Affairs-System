# 本地教务工具检查建议

## 文档目的
这份文档用于给明天继续接手的 Codex 作为检查反馈与优化建议。当前结论是：项目整体已经具备较完整的本地教务工具形态，功能覆盖面较大，自动化测试和前端构建也能通过；但在文件边界控制、下载安全约束和路由维护性方面，仍有几处值得优先处理的问题。

## 当前整体评价

### 优点
1. 项目整体方向基本符合最初目标：本地、单机、离线优先、中文后台系统。
2. 技术栈与结构比较合理：
   - 前端：Vue 3 + TypeScript + Vite
   - 后端：FastAPI + SQLAlchemy + SQLite
   - 目录结构清晰，前后端分离明显
3. 关键业务模块覆盖较完整，包括：
   - 基础数据
   - 学生与教师管理
   - 考试与成绩分析
   - 课表与工作量
   - 成长档案
   - 推荐中心
   - 评教与量化
   - 报表导出
   - 本地备份恢复
4. 当前验证结果正常：
   - 后端测试通过：16/16
   - 前端构建通过
5. 默认监听本地地址 `127.0.0.1`，符合本地工具的定位。

## 建议 Codex 优先处理的问题

### 1. 上传文件目录缺少边界校验
**优先级：高**

后端上传文件逻辑中，上传分类 `category` 被直接拼接到上传目录路径中使用。如果这个字段未严格限制，理论上可以通过类似 `../` 的写法把文件写到 `data/uploads/` 之外。

#### 风险说明
- 会破坏“附件必须保存在受控本地目录”的约束
- 可能导致文件写入到项目内的非预期目录
- 即使这是本地单机系统，也建议把文件存储边界收紧

#### 建议 Codex 处理方向
- 对 `category` 做白名单或安全字符校验
- 禁止 `..`、绝对路径、路径分隔符穿透
- 最终写入前校验目标路径必须位于 `settings.uploads_dir` 之内
- 为该问题补充自动化测试

---

### 2. 报表/下载接口对文件路径的约束还不够严格
**优先级：高**

部分下载逻辑会从数据库读取 `file_path`，再用 `settings.project_root / file_path` 组合出最终路径。如果数据库中出现绝对路径，Python 的路径拼接会直接采用绝对路径本身，导致下载边界变宽。

#### 风险说明
- 当前正常业务流程可能不会产生这个问题
- 但如果数据库出现脏数据、手动写入数据，或者未来新增导出逻辑复用了这套字段，下载接口可能被用来读取受控目录外的文件

#### 建议 Codex 处理方向
- 对报表、备份、模板、附件等所有下载接口做统一路径校验
- 不仅检查“文件是否存在”，还要检查“文件是否位于允许目录下”
- 可考虑封装统一的安全文件解析函数
- 为绝对路径、路径穿越、异常路径增加测试用例

---

### 3. 存在重复路由定义
**优先级：中**

发现 `/api/system/templates` 被两个不同文件重复定义。当前运行时似乎还能正常返回，但这种情况容易造成：
- 行为依赖注册顺序
- 后续维护时误改
- 文档与调试混乱

#### 建议 Codex 处理方向
- 删除重复接口定义，只保留一个权威实现
- 检查是否还有类似重复注册问题
- 保持系统路由职责边界清晰

---

### 4. 上传接口整文件读入内存，后续可做鲁棒性优化
**优先级：中 / 低**

目前多个上传接口使用整文件 `read()` 的方式读取内容。对于本地教务工具、小型 Excel 文件通常问题不大，但后续如果文件变大，可能带来不必要的内存压力。

#### 建议 Codex 处理方向
- 若当前阶段不急，可以先不改
- 后续可考虑：
  - 为上传文件大小增加限制
  - 对大文件采用更稳妥的处理方式
  - 在错误提示中明确超限原因

## 我建议 Codex 的修复顺序
1. 先修上传目录边界问题
2. 再统一收紧所有下载接口的文件路径校验
3. 清理重复路由
4. 视时间决定是否优化大文件上传策略

## 建议 Codex 修复后补的验证项
修复完成后，建议至少补以下验证：

### 自动化测试
1. 上传接口传入非法 `category` 时应返回错误
2. 上传接口传入正常 `category` 时应仍可正常保存
3. 下载接口遇到绝对路径记录时应拒绝访问
4. 下载接口遇到目录穿越路径时应拒绝访问
5. 现有备份、模板、报表、附件下载流程仍能正常通过

### 手工回归检查
1. 学生附件上传、下载、删除
2. 成长档案附件引用与下载
3. 模板下载
4. 报表导出与下载
5. 备份创建、下载、恢复
6. 系统设置页相关功能是否仍正常

## 给 Codex 的一句话总结
这个项目整体已经能用，重点不是推翻重做，而是**补齐文件上传与下载的安全边界**、**清理重复路由**、**增强系统层面的稳健性**。优先做小范围、可验证、低扰动的修复，不要为了“优化”而大改现有结构。
