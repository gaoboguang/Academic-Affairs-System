## 二、现在的技术形态

项目当前是“三层结构”：

1. 前端：Vue 3 + TypeScript + Vite  
   位置：`apps/frontend`

2. 后端：FastAPI + SQLAlchemy + SQLite  
   位置：`apps/backend`

3. 桌面壳：Electron  
   位置：`apps/desktop`

桌面版不是另做了一套系统，而是在现有前后端外面包了一层桌面壳。

## 四、后续还能继续加功能吗

能，而且现在的结构就是按“后续继续扩展”设计的。

原因很简单：

- 业务逻辑仍在 `apps/backend`
- 页面与交互仍在 `apps/frontend`
- 桌面层只负责启动和打包，不会锁死业务扩展

所以后续要新增模块、页面、接口、导入模板、报表、分析规则，都仍然可以继续往当前系统里加。

需要遵守的原则：

- 数据库结构变更要配 Alembic 迁移
- 核心规则不要硬编码在前端
- 附件继续走本地文件目录，不直接塞数据库

## 五、高校数据库后期应该放哪里

结论：**正式数据应该继续放 SQLite，不建议散落成独立 Excel 文件库。**

项目现在其实已经有高校数据库的模型，位置：

- `apps/backend/app/models/recommendation.py`

现有正式表包括：

- `college`
- `college_alias`
- `major`
- `college_major`
- `admission_record`

所以后续高校数据库的正确落点是：

1. 高校、专业、历年录取线、专业关系等结构化数据  
   放本地 SQLite

2. 原始导入文件  
   作为导入来源存在本地文件中，必要时保留在 `data/uploads/`

3. 导出报表  
   放 `data/exports/`

## 六、当前数据放置建议

浏览器版开发 / 本地运行：

- 主数据库：`data/app.db`

桌面版运行：

- 主数据库：Electron `userData/data/app.db`

也就是说，桌面版和开发版都还是“本地 SQLite + 本地文件目录”这一套思路，不需要单独维护一套外部数据库。
