## 1. 文档目标

本文档用于明确本地教务工具下一阶段的建设方向、系统边界、模块职责、数据模型、推荐与分析逻辑、开发优先级、实施里程碑和验收标准，供 Codex 按文档拆解开发任务并逐步落地。

这份文档不是单纯的需求清单，而是面向可开发、可迭代、可验收的实施文档。默认要求：

1. 开发结果必须可在本地离线环境运行。
2. 学生成绩等核心数据默认不出校。
3. 关键分析结论优先由规则计算与统计模型得出，本地模型负责解释增强，不直接替代核心计算。
4. 所有推荐、预测、分析结果必须可追溯到原始数据、计算过程和规则依据。
5. 所有新增模块应尽量复用当前技术栈、目录结构与现有业务底座，避免另起一套系统。

## 2. 项目定位与总原则

### 2.1 项目定位

本项目不是单纯的信息管理系统，也不是单纯的志愿填报系统，而是一个面向高中教学场景的“本地教务决策台”。

其核心价值不在于“功能多”，而在于以下两项核心能力：

1. **成绩分析能力**
   - 围绕每次考试形成多维度、可追溯、可对比的分析结果。
   - 同一份成绩数据可产出学生维度、教师维度、班级维度、年级维度等不同报告。
   - 支持历次考试趋势、进退步分析、预警分析、学科短板分析、阶段评价等。

2. **高考志愿与升学决策能力**
   - 基于学生成绩、位次、选科、学生意愿、升学路径偏好，输出可解释的志愿建议。
   - 以山东省为首期重点生源地，逐步扩展到全国。
   - 支持不同升学路径：普通夏季高考、春季高考、艺术类、体育类、强基计划、本科综评、专科综评、专项计划、单招、国际方向、港澳台方向等。

### 2.2 使用原则

1. **先服务自己，再考虑扩展**
   - 当前按个人使用设计，优先保证自己可长期稳定使用。
   - 暂不以全校协同、复杂权限、多租户为主要设计目标。
   - 但架构上应预留后期学校级扩展能力。

2. **本地优先、可靠优先**
   - 所有核心数据默认保存在本地 SQLite。
   - 本地模型可接入，但不得让模型成为唯一真相来源。
   - 优先保证规则可解释、计算可复核、数据可备份。

3. **引擎优先于页面**
   - 下一阶段不是盲目继续堆页面，而是优先补强数据底座、分析引擎、推荐引擎。
   - 页面只是引擎结果的呈现层。

4. **先山东，再全国**
   - 山东省作为首期重点场景，先做深做透。
   - 全国扩展采用“规则模板 + 省份差异补丁”的方式推进。

## 3. 当前项目基础与现实判断

### 3.1 当前已具备的能力基础

现有项目已经具备以下主干能力，可作为继续开发的稳定底座：

- 前端采用 Vue 3 + TypeScript + Vite + Element Plus
- 后端采用 FastAPI + SQLAlchemy + Pydantic + Alembic
- 数据库为 SQLite
- 已具备 Electron 桌面端第一版
- 已具备考试、成绩、学生、教师、成长档案、工作量、评教、报表等主工作流
- 已具备院校库、专业库、录取库、推荐历史、高考志愿工作台第一轮能力
- 已具备测试基础：Pytest、Vitest、Playwright

### 3.2 当前真正的核心缺口

下一阶段最关键的问题，不是“缺不缺模块”，而是“高价值模块的数据底座和正式建模是否到位”。

当前最重要的缺口主要集中在以下几个方面：

1. 正式的分数—位次映射表尚未补齐
2. 赋分规则尚未形成独立可维护体系
3. 选科要求尚未形成统一编码层
4. 省份规则仍是第一轮基线，缺少更多省份/年份的精细化规则
5. 高考推荐目前已有可用主链路，但底层正式数据模型仍需加厚
6. 成绩分析虽然已有主链路，但缺少更完整的指标树、统一分析引擎与报告模板体系

### 3.3 当前阶段判断

项目当前可判断为：

- 主业务骨架已完成
- 核心数据流已打通
- 已达到“可演示、可迁移、可继续开发”的阶段
- 下一阶段应进入“引擎深化、数据补齐、产品化收口”的阶段

## 4. 下一阶段总目标

下一阶段总目标分为一个总目标和四个阶段性目标。

### 4.1 总目标

将本地教务工具建设为一套以“成绩分析 + 高考志愿”为双核心的本地决策系统，使其满足以下要求：

- 考试数据可快速导入
- 多维成绩分析可自动生成
- 高考志愿推荐可基于正式数据与规则进行计算
- 推荐结果可解释、可调整、可导出
- 数据与模型均可在本地独立运行
- 系统可作为后续对外演示和学校级封装的核心产品底座

### 4.2 阶段性目标

#### 目标 A：统一成绩分析引擎

实现统一的成绩分析指标树、规则引擎和报告生成引擎，使系统能按学生、教师、班级、年级四大维度稳定输出分析结果。

#### 目标 B：补齐山东志愿数据底座

完成山东省首期所需的数据底座，包括院校主档、专业主档、招生计划、历年录取、分数—位次、一分一段、选科要求、批次规则、升学路径规则等。

#### 目标 C：升级高考志愿推荐引擎

将当前推荐从“第一轮可用状态”升级为“正式数据驱动状态”，形成可解释、可配置、可追溯的推荐体系。

#### 目标 D：形成可交付产品形态

完善文档、导入模板、报表、备份恢复、数据修复、演示路径，为后续自用稳定运行和对外展示做准备。

## 6. 总体架构建议

建议继续沿用当前 Monorepo 结构，不重建项目。

### 6.1 推荐的逻辑分层

#### 表现层

- Web 前端（Vue 3）
- Electron 桌面端
- 报表打印页
- 导出任务页

#### 接口层

- FastAPI 路由
- DTO / Schema 校验
- 统一错误处理
- 统一权限预留接口

#### 业务层

- 学生服务
- 考试与成绩服务
- 成绩分析服务
- 报表服务
- 高考推荐服务
- 数据采集导入服务
- 规则服务
- 本地模型服务

#### 引擎层

- 成绩指标计算引擎
- 成绩趋势分析引擎
- 预警规则引擎
- 高考升学路径识别引擎
- 志愿推荐与排序引擎
- 报告文本生成引擎

#### 数据层

- SQLite 业务库
- 文件目录（导入模板、导出文件、附件、备份）
- 中间数据目录（采集结果、标准化结果、待审核文件）

### 6.2 为什么要强调“引擎层”

因为后续无论是学生报告、教师报告、班级报告、年级报告，还是高考推荐、升学路径建议，本质都不是页面逻辑，而是引擎逻辑。

如果引擎层不独立：

- 逻辑会散落在页面和接口中
- 难以维护
- 难以复用
- 难以测试
- 难以接入本地模型进行解释增强

因此必须把“可计算、可解释、可测试”的逻辑统一沉到服务层与引擎层。

## 7. 成绩分析体系设计

成绩分析是本系统第一核心。

### 7.1 分析输入

考试成绩来源采用手动上传，统一走 Excel 导入。

每次考试建议最少采集以下字段：

- 学年
- 学期
- 考试名称
- 考试类型（联考模拟 / 期末 / 期中 / 月考 / 平时测试）
- 考试时间
- 年级
- 班级
- 学生学号
- 学生姓名
- 科目
- 原始分
- 满分
- 是否缺考
- 是否违纪
- 排名口径（班级/年级）
- 数据批次号

### 7.2 考试类型权重

系统需支持不同考试类型权重，默认建议如下：

1. 联考模拟：最高权重
2. 期末成绩：较高权重
3. 期中成绩：中高权重
4. 月考成绩：中等权重
5. 平时成绩：较低权重

实现要求：

- 权重可配置
- 支持全局默认值
- 支持年级级别覆盖
- 支持算法版本管理

### 7.3 统一指标树

必须先定义统一指标树，再做报表。

#### 学生维度核心指标

1. 本次考试总分
2. 本次考试单科成绩
3. 班级排名
4. 年级排名
5. 班级名次变化
6. 年级名次变化
7. 各科进退步幅度
8. 历次考试趋势
9. 优势学科
10. 薄弱学科
11. 偏科程度
12. 稳定性指数
13. 波动性指数
14. 高考预估分区间
15. 高考预估位次区间
16. 风险预警标签
17. 综合评价结论
18. 选科倾向判断
19. 升学路径适配度
20. 志愿推荐输入摘要

#### 教师维度核心指标

1. 所任班级总分表现
2. 所任学科均分
3. 及格率
4. 优秀率
5. 高分率
6. 低分率
7. 学生成绩分层分布
8. 进步学生数
9. 退步学生数
10. 风险学生数
11. 学科贡献分析
12. 班级横向对比
13. 同学科横向对比

#### 班级维度核心指标

1. 班级总分均分
2. 班级总分排名
3. 班级单科均分
4. 班级各分数段人数
5. 班级优秀率
6. 班级及格率
7. 班级进步率
8. 班级退步率

13. 班级整体评价

#### 年级维度核心指标

1. 年级总分分布
2. 年级单科分布
3. 上线率
4. 目标达成率
5. 进步率
6. 退步率
7. 班级横向对比
8. 学科横向对比

11. 年级管理建议

### 7.4 分析报告类型

系统至少要支持以下报告类型：

1. 学生个人成绩分析报告
2. 教师任教分析报告
3. 班级成绩分析报告
4. 年级成绩分析报告
5. 单科质量分析报告
6. 风险预警名单报告
7. 学科攻坚报告
8. 选科建议报告
9. 升学路径建议报告
10. 志愿推荐报告

### 7.5 报告生成机制

建议采用“两段式生成”：

#### 第一段：确定性计算层

由后端规则/算法完成：

- 分数统计
- 排名统计
- 趋势计算
- 波动计算
- 预警判定
- 预估分与预估位次
- 推荐结果排序

#### 第二段：解释生成层

由模板或本地模型完成：

- 结论表达
- 亮点总结
- 问题归纳
- 复习建议
- 备考建议
- 志愿说明

要求：

- 模型不得修改关键数字
- 模型不得自行编造排名和概率
- 所有结论要可回溯到指标输入

### 7.6 高考成绩预估逻辑

需支持至少两种模式：

1. **系统预估模式**
   - 根据历次考试成绩与权重计算预估分
   - 输出区间而不是单点值
   - 优先采用最近考试加权

2. **学生自填模式**
   - 学生可手动填写自己预估高考分数
   - 若学生主动填写，则推荐计算优先使用学生自填值
   - 同时保留系统预估值供对比

建议输出：

- 系统预估分
- 系统预估分区间
- 学生自填预估分
- 推荐计算采用分数
- 推荐计算采用原因

### 7.7 学习倾向与选科建议

在学生分科前，系统需要根据历次成绩分析学习倾向，形成选科建议。

建议分析维度：

- 学科长期表现
- 学科稳定性
- 学科成长速度
- 优势学科组合
- 弱势学科组合
- 目标院校/专业对选科的要求
- 学生兴趣偏好
- 后续升学路径约束

建议结果输出：

- 建议优先选科组合
- 不建议组合
- 推荐原因
- 风险说明
- 与目标专业匹配度

## 8. 高考志愿与升学体系设计

高考志愿是本系统第二核心。

### 8.1 推荐总体原则

1. 以成绩/位次为核心依据
2. 以最新考试成绩为最大权重
3. 学生志愿偏好用于排序修正，不替代成绩主逻辑
4. 升学路径需先判断，再进入各自子推荐链路
5. 推荐必须可解释
6. 推荐必须保留规则依据和数据来源摘要

### 8.2 支持的升学路径

山东首期建议支持：

1. 普通夏季高考
2. 春季高考
3. 艺术类
4. 体育类
5. 强基计划
6. 本科综合评价
7. 专科综合评价
8. 高校专项 / 地方专项 / 国家专项
9. 单独招生
10. 国际方向
11. 港澳高校方向
12. 其他特殊类型招生

### 8.3 推荐流程

#### 步骤 1：确定推荐输入

输入来源包括：

- 生源地
- 目标年份
- 当前最新成绩
- 历次成绩加权结果
- 学生自填预估分
- 系统预估分
- 当前位次或估算位次
- 学生类别
- 艺体方向
- 选科情况
- 学生偏好

#### 步骤 2：确定升学路径池

根据学生条件、成绩和意愿，判定适合进入哪些升学路径：

- 普通类
- 艺术类
- 体育类
- 强基/综评/专项
- 专科/高职
- 国际/港澳台

#### 步骤 3：规则过滤

按以下规则过滤：

- 生源地匹配
- 年份匹配
- 批次匹配
- 选科要求匹配
- 学生类别匹配
- 艺体类别匹配
- 升学路径匹配
- 特殊计划条件匹配

#### 步骤 4：风险分层

形成冲、稳、保分层：

- 冲：存在一定挑战但可尝试
- 稳：较高匹配度
- 保：相对稳妥

#### 步骤 5：偏好修正排序

按以下因素做排序修正：

- 目标地区
- 院校偏好
- 行业偏好
- 专业偏好
- 是否服从调剂
- 学费敏感度
- 就业方向匹配度

#### 步骤 6：结果解释输出

每个候选结果需要给出：

- 推荐原因
- 录取参考依据
- 风险说明
- 选科说明
- 专业/行业匹配说明
- 是否满足学生偏好

### 8.4 推荐引擎必须保留的输入

推荐结果必须保存以下上下文，便于追溯：

- 推荐时使用的分数
- 推荐时使用的位次
- 推荐时使用的考试依据
- 推荐时使用的算法版本
- 推荐时使用的规则版本
- 推荐时的偏好输入
- 推荐生成时间
- 推荐结果解释摘要

### 8.5 本地模型在志愿中的角色

## 22. 风险与应对

### 22.1 主要风险

1. 规则写得多但数据底座不完整，导致推荐看起来很多但不可信
2. 本地模型接入过早，导致关键逻辑不可解释
3. 数据抓取过散，维护成本过高
4. 页面开发过快，引擎抽象不足
5. 报告越来越多，但底层指标不统一
6. 多条并行开发线同时改文档与代码，容易出现口径漂移

### 22.2 应对策略

1. 先补底座，再扩解释
2. 先做规则计算，再接模型增强
3. 先山东做深，再全国扩展
4. 先统一指标，再做多报告模板
5. 所有核心算法和规则版本化
6. 数据库章节与非数据库章节分线维护，合并前做文档口径校对

## 26. 文档结论

下一阶段的工作重点，必须从“继续加模块”转到“补强数据底座和分析/推荐引擎”。

就当前仓库状态而言，这件事已经不是抽象方向，而是两个并行动作：

1. 数据库与正式建模继续由数据库线推进
2. 非数据库主线继续把分析引擎、推荐解释、打印导出、桌面体验和验证体系补到一致

这套系统真正的壁垒，不是做多少页面，而是：

1. 能不能把成绩分析做成真正可靠的多维报告引擎
2. 能不能把高考志愿做成真正基于正式数据与规则的推荐引擎
3. 能不能在本地离线环境下，把规则计算、本地模型解释、导出报表三者稳定结合

只要这三点做成，这个系统不仅能长期自己使用，也具备后续向学校级产品扩展的基础。
